# 受注管理システム開発・改修規約

このスキルは、AIエージェントが本システムのコアロジックやデータベース構造を不適切に破壊することを防ぎ、プロジェクトの長期的な安定性を保つために存在します。

## 1. 原則 (Core Principles)
- **直接編集の制限**: `app/models/` および `app/services/` 以下の重要なバックエンドファイルを、ユーザー（人間）の明示的なチャットでの承認なしに書き換えてはなりません。
- **アーキテクチャの遵守**: 
    - データベース操作は必ず `app/models/` に定義し、コード内に生のSQLを埋め込むことは禁止です。
    - ビジネスロジック（在庫計算、状態遷移バリデーション）は必ず `app/services/` に実装し、テンプレート（HTML）やAPIエンドポイント（main.py）に複雑なロジックを記述しないでください。
- **日本語インターフェースの維持**: UIのラベルやステータス表示は日本語を基本とします。

## 2. 改修・開発プロセス (Standard Workflow)
エージェントは、機能追加やバグ修正を行う際、以下のステップを厳守しなければなりません。

1. **現状分析と提案**: 
    - 関連するファイルのコードを読み、`implementation_plan.md` に変更案（Before/After）を記述する。
    - 変更が既存の受注データや在庫整合性に与える影響を評価し、報告する。
2. **ユーザー承認の獲得**: 
    - チャットを通じて設計案を示し、「実行して良いか」をユーザーに問う。
3. **実行と検証**: 
    - ユーザーの承認を得た後、最小限の単位でコードを修正する。
    - 修正後、サーバーの再起動または既存機能の動作確認を行い、結果をエビデンス（ログ等）と共に報告する。

> [!NOTE]
> Git/GitHubを使用したブランチ作成、コミット、PR運用などの開発フローについては `git-workflow` スキルを参照してください。

## 3. 保護対象の重要ファイル (Protected Files)
- `app/database.py`: 接続設定とセッション管理。
- `app/models/models.py`: データベース構造（テーブル定義）。
- `app/services/services.py`: コア・ビジネスロジック（受注・在庫処理）。

## 4. MCPサーバーの活用 (Optional but Recommended)
利用可能な場合、開発支援用MCPサーバー（`mcp/`）のツールを使用して、コードの整合性チェックや影響分析の補助を受けることが推奨されます。
## 5. UI/UX デザイン基準 (Design Standards)
本プロジェクトでは、プレミアムなユーザー体験を提供するため以下の基準を厳守します。

- **テーマ**: Soft Modern / Glassmorphism (Luminous Clarity)
- **タイポグラフィ**: 
    - Display: `Outfit` (見出し、アクセント)
    - Body: `Plus Jakarta Sans` (本文、入力フォーム)
- **カラーパレット**: 
    - 背景: `radial-gradient(circle at top right, #eef2ff, #fdf2f8, #f8fafc)`
    - アクセント: Indigo (`#6366f1`), Rose (`#f43f5e`), Emerald (`#10b981`)
- **コンポーネント形状**:
    - ガラスカード: `bg-white/70`, `backdrop-blur`, `border-white/30`
    - 角丸: カードは `2.5rem` 〜 `3rem`, ボタン・入力欄は `1.5rem`
- **ナビゲーション**: 固定式の左サイドバーレイアウト (幅: `w-64`) を維持してください。
- **言語設定**: すべてのUIラベル、メッセージ、プレースホルダーは**日本語**で記述してください。
